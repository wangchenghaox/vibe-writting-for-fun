# 需求确认

在正式写章节、做人设、做大纲、改写、审稿或保存前，先确认用户真正想要的方向，并给出 1-3 个可行方案帮助用户选择。

## 适用场景

- 用户要求开始章节、人物、大纲、故事、世界观、剧情、审稿或改写任务，但方向、范围、风格或保存目标不够明确。
- 用户只说“帮我写/设计/想一个”时，需要先收束题材、目标和输出形态。
- 用户要求保存，但当前对话没有完整可保存内容、目标文件路径或覆盖确认。

## 角色边界

- 主 Agent：直接面向用户使用本 skill，负责澄清、收束方案和确认保存意图；确认后再把具体生成、审稿、改写、整理成稿或文件修改委派给子 Agent。
- 子 Agent：不能向用户提问或等待输入；如果被分配的任务信息不足，返回“缺口清单 + 建议主 Agent 追问的问题”，不要继续猜测高成本内容。
- 本 skill 不负责生成完整正文、大纲、人设、审稿意见或执行保存。

## 上下文读取

- 文件路径使用当前 sandbox 相对路径，不要加 `novels/` 前缀。
- 如用户提到当前小说，优先定位 `{novel_slug}/` 下的文件。
- 如用户要求基于已有大纲、章节、角色或审稿文件，可用 `list_files`、`search_files` 定位 Markdown 文件，再用 `read_file` 读取。
- 默认查找路径是 `{novel_slug}/` 下的 `.md` 文件。
- 未确认需求前，不要使用任何保存工具；本 skill 本身不暴露保存工具，也不调用小说/文档领域工具。

## 工作流程

1. 先用 1-2 句复述你理解的用户目标，标出不确定点。
2. 同时给出 1-3 个可行方案供用户参考；每个方案说明风格、重点和适合的结果。
3. 只问关键问题，通常 3-5 个以内，优先问会改变输出方向的问题。
4. 如果用户已经给足信息，也要先简短确认方向，并问“按这个方向开始吗？”
5. 用户确认后，主 Agent 不直接生成成品；改为给对应执行 skill 创建子 Agent 任务单。

## 确认维度

- 任务类型：章节正文、章节细纲、总纲、角色卡、关系网、审稿、改写或保存。
- 输出范围：第几章、多少字、覆盖哪些角色或剧情段。
- 创作方向：题材、基调、爽点、悬疑点、冲突重点。
- 风格要求：热血、克制、黑暗、轻松、悬疑、史诗、都市感等。
- 上下文来源：需要读取哪些大纲、章节、角色卡或审稿文件。
- 保存意图：是否保存，保存到哪个 `.md` 文件路径，是否允许覆盖旧内容。
- 验收标准：用户希望这次结果达成什么效果。

## 输出要求

- 使用“我理解的是...”开头，快速复述需求。
- 接着列出“可选方向”，给 1-3 个方案，不要超过 3 个。
- 最后列出“需要你确认”，只问关键问题。
- 先确认再创作；用户未确认前，不进入完整正文、大纲、人设或审稿输出。

## 质量标准

- 方案要彼此有明显差异，不要只是换说法。
- 问题要能减少返工，避免问无关细节。
- 如果用户已经表达清楚，不要过度盘问，给简短确认即可。
- 确认结果要足够写成子 Agent 任务单。

## 常见误区

- 没确认需求就直接开始写章节、人设、大纲或审稿意见。
- 主 Agent 在用户确认后直接输出成品，而不是委派子 Agent。
- 给太多方案让用户难以选择。
- 只问问题，不给任何可参考方向。
- 未确认不调用保存工具。
- 把确认环节写成长篇分析，拖慢创作开始。
