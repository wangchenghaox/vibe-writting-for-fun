# 人物设定

把角色设计成能推动情节、制造关系张力并持续变化的人。此 skill 同时服务主 Agent 的任务编排和子 Agent 的人物设定产出。

## 适用场景

- 用户要求设计主角、配角、反派、人物关系、角色卡或成长弧。
- 用户要求补强角色动机、弱点、关系冲突或人物小传。
- 不适用于直接写章节正文；角色设定完成后可交给 chapter-writer 使用。

## 角色边界

- 主 Agent：只确认需求、整理约束、列简短计划，并调用 `create_sub_agent` 委派；不要直接输出完整角色卡、人设或调用保存工具。
- 子 Agent：执行任务单，读取资料、设计人物、按要求保存；不能向用户提问或等待输入，信息不足时把缺口和建议下一步写回执行结果。
- 主 Agent 委派时，`task` 必须写清：角色功能、故事题材、已有设定路径、需要设计的角色范围、输出结构、是否保存、目标路径、是否允许并入/覆盖已有文件、验收标准。
- 如果用户只给“帮我设计一个角色”这类开放请求，先用 requirement-confirmer 收束角色功能和故事方向。

## 上下文读取

- 正式设计、细化、并入或保存角色设定前，先按 progress-summarizer 读取小说根目录的 `progress.md`，确认当前进度、角色状态和待确认问题。
- 如果 `progress.md` 不存在，只做轻量目录检查，不要逐个读取所有文件；角色设定生成、修改或保存后必须更新进度总结。
- 主 Agent 委派任务单应写清 `progress.md` 路径；子 Agent 返回结果时说明是否已更新进度总结。
- 文件路径使用当前 sandbox 相对路径，不要加 `novels/` 前缀。
- 如用户指定小说，优先定位 `{novel_slug}/` 下的 Markdown 文件。
- 如角色需要参考已有大纲，使用 `search_files` 或 `list_files` 查找 `{novel_slug}/outlines/*.md`，再用 `read_file` 读取。
- 如角色需要参考已有角色文件，查找 `{novel_slug}/characters/*.md`。
- 如果已有设定不足以判断角色定位，子 Agent 返回缺口，不要直接向用户提问。

## 工作流程

1. 明确角色功能：推动主线、制造冲突、提供情感牵引、承担主题或作为反面镜像。
2. 写角色卡：身份、外在目标、内在缺口、欲望、恐惧、能力边界、代价、秘密。
3. 设计行为逻辑：角色遇到压力时会如何选择，为什么这个选择会推动剧情。
4. 设计关系网：说明每段关系的利益、情感、误解或价值观冲突。
5. 设计成长弧：起点缺陷、关键打击、错误选择、转变节点、结局状态。
6. 设计完成后，如果任务单要求先出方案，返回角色方向摘要和建议主 Agent 请用户确认的点；不要直接问用户。
7. 如果任务单已明确继续扩展小传、并入大纲或保存，子 Agent 直接完成。
8. 如果要求保存角色卡或把人设并入大纲，先读取目标文件，再用文件工具写入或修改。
9. 修改或生成新内容后，更新 `progress.md`，记录角色状态、影响文件、待确认问题和下一步。

## 保存规则

- 保存角色卡时只使用 `write_file` 或 `edit_file`，不要使用领域保存工具。
- 角色卡默认写入 `{novel_slug}/characters/{character_id}.md`，这是当前 sandbox 相对路径，不要加 `novels/` 前缀。
- 并入大纲时先用 `read_file` 读取原大纲，再用 `edit_file` 合并修改；如果会覆盖或改动既有设定，先询问用户确认应由主 Agent 完成。
- 不要只保存新增片段导致旧大纲丢失。
- 文件类型必须是 Markdown，扩展名必须是 `.md`。
- 用户没有明确要求保存时，只输出角色设定，不调用保存工具。
- 用户确认前不要保存，也不要覆盖已有 outline。

## Markdown 固定格式

```markdown
---
type: character
id: {character_id}
name: {character_name}
role: protagonist|antagonist|supporting
status: draft
---

# 角色：{character_name}

## 角色定位

## 外在目标

## 内在缺口

## 能力边界

## 关系张力

## 成长弧
```

## 输出要求

- 输出至少包含“角色定位、外在目标、内在缺口、能力边界、关系张力、成长弧”。
- 反派要写清自己的正当性、底线和失败风险。
- 人物关系网要说明张力来源，不只列名字。
- 如需确认，结尾写“建议主 Agent 确认：...”而不是直接问用户。

## 质量标准

- 人物不能只是一组标签，要有会驱动行动的欲望。
- 强角色要有弱点，弱角色要有选择。
- 反派应有自己的逻辑，而不只是阻碍主角。
- 角色设定要能转化成章节中的行动、冲突和选择。

## 常见误区

- 主 Agent 读取本 skill 后直接写出完整角色卡。
- 只写外貌、身份和性格标签，没有欲望和行动逻辑。
- 把反派写成纯恶阻碍物，没有自己的收益和恐惧。
- 关系网只是阵营表，没有情感或利益张力。
- 未读取原大纲就声称已经并入。
- 未经用户确认就覆盖 outline。
