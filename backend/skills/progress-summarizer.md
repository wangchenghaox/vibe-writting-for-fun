# 进度总结

维护小说根目录的 `progress.md`，让写作 Agent 进入工作前先知道当前状态，完成修改后留下可靠的进度记录。

## 适用场景

- 正式开始写作、整理、改写、审稿、保存或文件修改前。
- 子 Agent 完成大纲、章节、角色、审稿意见、设定资料等内容生成或修改后。
- 用户要求查看当前进度、总结已完成内容、确认下一步或恢复中断工作时。

## 角色边界

- 主 Agent：在任务单中写清需要读取和更新 `progress.md`，并说明小说根目录或目标文件路径。
- 子 Agent：执行任务前读取 `progress.md`；完成生成、修改或保存后更新 `progress.md`；如果信息不足，记录缺口并交回主 Agent。
- 不要把 `progress.md` 当成完整资料库；它只保存摘要、变更和下一步，不保存长篇正文或完整大纲。

## 上下文读取

- `progress.md` 固定放在小说根目录，例如 `{novel_slug}/progress.md`；如果当前工作目录已经是小说根目录，则路径是 `progress.md`。
- 工作开始时先读取 `progress.md`。如果不存在，先用 `list_files` 或 `search_files` 做轻量目录检查，再创建初始总结。
- 不要逐个读取所有文件；只读取任务明确需要的章节、大纲、角色或审稿文件。
- 如果任务单指定了目标文件，读取 `progress.md` 后再读取目标文件或相邻必要文件。

## 工作流程

1. 定位小说根目录和 `progress.md` 路径。
2. 使用 `read_file` 读取 `progress.md`；文件不存在时，用 `list_files` 查看根目录结构和关键子目录。
3. 提取当前状态、已完成内容、最近变更、待确认问题和下一步。
4. 执行具体写作、整理、改写、审稿或保存任务。
5. 修改或生成新内容后，更新进度总结，写清本次变更、影响文件、未解决问题和建议下一步。
6. 如果本次没有实际改动，只在用户要求总结时更新；不要为了空操作反复改写 `progress.md`。

## 写入规则

- 新建 `progress.md` 时使用 `write_file`，已存在时优先使用 `edit_file` 更新相关段落。
- 不要覆盖掉已有历史状态；更新“最近变更”和“下一步”时保留仍然有效的信息。
- 如果目标内容较乱，可以用 `write_file` 重写完整 `progress.md`，但必须保留仍有效的事实。
- 写入内容必须是完整 Markdown，不要只写一句说明。

## Markdown 固定格式

```markdown
# 当前进度

## 项目状态

## 已完成内容

## 最近变更

## 待确认问题

## 下一步
```

## 输出要求

- 总结要短，优先记录事实、文件路径、状态和下一步。
- 最近变更写清“做了什么”和“影响哪些文件”。
- 待确认问题只记录会影响后续工作的关键问题。
- 返回执行结果时简要说明 `progress.md` 已读取或已更新，以及更新了哪些要点。

## 质量标准

- 进度总结能让下一次 Agent 不读全量文件也知道从哪里继续。
- 记录必须和实际文件改动一致，不要把未完成事项写成已完成。
- 路径使用当前 sandbox 相对路径，不要加 `novels/` 前缀。
- 内容保持中文、简洁、可执行。

## 常见误区

- 没读 `progress.md` 就开始续写或改稿。
- 修改或生成新内容后忘记更新进度总结。
- 为了总结当前状态而逐个读取所有文件。
- 把完整章节、完整大纲或长篇审稿意见粘进 `progress.md`。
- 把用户闲聊、临时情绪或敏感信息写入进度总结。
